Visão Geral do Sistema

O sistema é composto por dois processos independentes que se comunicam via HTTP/REST com autenticação JWT. O frontend (Next.js) roda na porta 3000 e o backend (FastAPI) na porta 8000.

🖥️ Frontend — Next.js 14 · React 18 · TypeScript · Tailwind CSS
Páginas, componentes, gráficos (Recharts), formulários (Zod + react-hook-form), SWR para cache de dados · localhost:3000
↕ HTTP / REST + Bearer JWT
⚙️ Backend — FastAPI · Pydantic v2 · SQLAlchemy 2 · JWT + bcrypt
9 routers · autenticação · validação · ORM · cálculos financeiros · IA por pattern matching · localhost:8000
↕ SQL (ORM)
🗄️ Banco de Dados — SQLite (dev) → PostgreSQL (produção)
5 tabelas relacionais · cascade delete · isolamento por user_id em toda query
💡
Princípio de isolamento Cada endpoint autenticado filtra dados por current_user.id. Nenhum usuário acessa dados de outro — garantia aplicada no nível da query SQLAlchemy, não apenas na lógica de negócio.

Mapa de Endpoints da API

FastAPI gera documentação OpenAPI automaticamente em http://localhost:8000/docs. Todos os endpoints (exceto /auth/register e /auth/login) exigem Bearer Token JWT.

Auth

POST /auth/register Cria conta + 16 categorias padrão → 201 Token + User
POST /auth/login Autentica email/senha → 200 Token + User
GET /auth/me Dados do usuário autenticado → UserOut

Categorias

GET /categories?type=expense|revenue Lista categorias do usuário filtradas por tipo
POST /categories Cria nova categoria → 201 CategoryOut
PUT /categories/{id} Atualiza nome, cor ou ícone
DELETE /categories/{id} Remove categoria → 204 No Content

Despesas

GET /expenses?month=2024-06&category_id=5 Lista com filtros: mês, intervalo de datas, categoria
POST /expenses Cria despesa → 201 ExpenseOut (com eager load da categoria)
GET /expenses/{id} Detalhe de uma despesa
PUT /expenses/{id} Atualização parcial (campos opcionais)
DELETE /expenses/{id} Remove → 204 No Content

Receitas

GET /revenues?month=2024-06 Lista receitas com filtros equivalentes ao de despesas
POST /revenues Cria receita → 201 RevenueOut
PUT /revenues/{id} Atualização parcial
DELETE /revenues/{id} Remove → 204 No Content

Metas

GET /goals Lista metas com campo calculado percentage
POST /goals Cria meta → 201 GoalOut
PUT /goals/{id}/add-amount Adiciona valor à meta — não ultrapassa target_amount
DELETE /goals/{id} Remove meta → 204 No Content

Dashboard & Análise

GET /dashboard/summary?month=2024-06 KPIs: saldo, receitas, despesas, lucro, top_expenses, expense_categories
GET /dashboard/charts 4 datasets dos últimos 6 meses para os gráficos
GET /reports/summary?period=monthly&reference_date=2024-06-01 Relatório comparativo com período anterior + lista de transações
POST /ai/chat Chat com análise de padrões financeiros → response Markdown
GET /radar/analysis Score 0–100, alertas, detecção de desperdícios, previsões

Fluxo de uma Transação (CRUD)

Ciclo completo de uma operação de despesa desde o preenchimento do formulário até o rerender da tabela via SWR.

1

Usuário preenche ExpenseForm

Campos: descrição, categoria, valor, data, método de pagamento, observação (opcional)

2

Validação Zod (client-side)

Campos obrigatórios, amount positivo, date válida. Erros exibidos inline antes de enviar.

3

POST /expenses com Bearer Token

lib/api.ts injeta o token do localStorage no header Authorization: Bearer ...

4

Backend valida e persiste

HTTPBearer autentica → Pydantic valida body → SQLAlchemy INSERT → db.refresh() → eager load category

5

201 ExpenseOut retornado

Inclui id, created_at e objeto category aninhado (joinedload).

6

SWR mutate → rerender automático

Frontend invalida o cache /expenses?month=..., SWR refetch e a tabela atualiza sem reload.

Radar Financeiro — Score 0–100

O Radar calcula automaticamente a saúde financeira do usuário ao chamar GET /radar/analysis. O score é composto por 4 dimensões e gera alertas, previsões e detecção de desperdícios.

Dimensões do Score

0–40 pts
Expense Ratio
(1 − despesas/receitas) × 40
0–20 pts
Goal Tracking
+20 se tem metas cadastradas
0–20 pts
Revenue Sources
5 pts por categoria de receita diferente
0–20 pts
Expense Stability
Estabilidade vs mês anterior

Checklist de Segurança em Produção

  • 🔑 Trocar SECRET_KEY por string aleatória de 64+ caracteres — nunca usar o valor padrão de desenvolvimento
  • 🐘 Migrar DATABASE_URL para PostgreSQL com SSL (?sslmode=require) em produção
  • 🌐 Atualizar allow_origins no CORS para o domínio real — remover localhost
  • 🔒 HTTPS obrigatório — configurar certificado TLS/SSL (Cloudflare ou Let's Encrypt)
  • 🛡️ Rate limiting nos endpoints /auth/login e /auth/register para prevenir brute force
  • 📦 Backup automático diário do banco de dados com retenção de 30 dias
  • 📊 Monitoramento de erros e performance (Sentry, Datadog ou similar)
  • 🔄 Variáveis de ambiente nunca no repositório Git — usar secrets do CI/CD (GitHub Actions, Railway)
✅
Pronto para produção O sistema já implementa as práticas essenciais: hash bcrypt, JWT com expiração, isolamento por user_id, validação em duas camadas (Pydantic + Zod) e CORS configurado. Os itens do checklist acima são ajustes de ambiente, não mudanças de código.
